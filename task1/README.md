# Task 1 — Multi-agent orchestration

Four agents — **Supervisor**, **Inventory**, **Production**, **Report** — coordinate to answer a natural-language objective about a cosmetics manufacturer's operations. The hard architectural question the brief flags is "is this a multi-agent system or a pipeline?". My answer is in the contract.

## How agents communicate

Every inter-agent call is an [`AgentMessage`](../shared/messages.py) — a pydantic model carrying `trace_id`, `from_agent`, `to_agent`, `intent`, a typed `payload`, `status`, `attempts`, `metadata` (tokens/latency/cost), and an optional structured `AgentError`. Never raw strings.

The per-intent payload shapes (`CheckStockInput`, `CheckScheduleOutput`, `SupervisorPlan`, etc.) live in [shared/intents.py](../shared/intents.py) and are registered in `INTENT_INPUT_SCHEMA` / `INTENT_OUTPUT_SCHEMA`. The base class [`Agent`](../shared/agent_base.py) auto-validates inputs **before** delegating to the subclass and outputs **after** — so an agent that hallucinates a malformed response gets caught here, not 30 lines later in the supervisor.

What makes this not a pipeline:

- Each specialist runs its own LLM call with its own system prompt and validates its own output.
- The supervisor does not know HOW any specialist reasons — it only knows the intents they expose.
- Failures come back as typed `AgentError` envelopes, not exceptions bubbling up.
- The supervisor's planning, retry, and synthesis are three distinct LLM calls with three distinct prompts, not one mega-prompt.

## How failures are handled

| Failure mode | Detection | Recovery |
|---|---|---|
| Invalid input payload | base class validates against `INTENT_INPUT_SCHEMA` | returns `AgentError(error_class="schema_violation", retryable=False)` — supervisor does not retry |
| Hallucinated / malformed output | base class validates against `INTENT_OUTPUT_SCHEMA` | returns `schema_violation, retryable=True` — supervisor retries with a *modified* payload generated by another LLM call (see `SupervisorAgent._revise_payload`) |
| LLM exception (JSON parse / API error) | caught in base class | returns `AgentError(error_class="llm_error", retryable=True)` |
| Tool / upstream down | specialist returns `AgentError(error_class="upstream_unavailable")` | supervisor retries up to `SUPERVISOR_MAX_RETRIES` (default 1), then **escalates** the subtask and continues with the others |
| Unknown agent in plan | supervisor checks the registry | subtask marked `error / unknown_agent` immediately |

The "retry with a *modified* instruction" — Task 1 hard requirement #2 — is implemented as a small LLM call seeded with the original payload + the error message, returning a corrected payload. This is testable: see `tests/test_task1_orchestration.py::test_supervisor_retries_hallucinated_output_and_recovers`.

Graceful degradation: when a subtask escalates, the supervisor does NOT abort. It records the failure in the trace, includes "this subtask was escalated" in the synthesis prompt, and produces the best answer it can from the data that succeeded.

## Logging

Every agent call, every tool call, and every LLM call is wrapped in [`shared/trace.py`](../shared/trace.py) and lands in the `traces` SQLite table with `timestamp, actor, actor_kind, input, output, tokens, latency_ms, cost_usd, status, error_class`. The `parent_span_id` chain reconstructs the full call tree for any `trace_id`. This single table is what Task 4C's dashboard queries.

## Eval harness — 5 scenarios and why

Run: `python -m task1.eval.runner` (requires `GROQ_API_KEY` in `.env`).

| # | Scenario | What it proves |
|---|---|---|
| s1 | "Are we running low on glycerin or vitamin E?" | **Routing precision** — only `inventory` should be called. `production` and `report` must NOT appear. Catches over-routing. |
| s2 | "What's blocking production today?" | **Routing in the other direction** — only `production`. Catches a bias toward calling inventory reflexively. |
| s3 | Full morning briefing (stock + bottlenecks) | **Fan-out + synthesis** — inventory + production + report all run, supervisor synthesises. |
| s4 | Full briefing but `production` is replaced with `AlwaysFailingProduction` | **Graceful degradation** — production must be `escalated`, inventory must still `success`, supervisor must still produce a final answer (Task 1 req #4). |
| s5 | Stock query but `inventory` is replaced with `HallucinatingInventoryOnce` (malformed payload on call 1, valid on call 2) | **Retry-with-modified-instruction** — proves the recovery loop works without manual intervention (Task 1 req #2). |

Each scenario asserts:
1. Expected agents were invoked (queried from the `traces` table for the scenario's `trace_id`).
2. Forbidden agents were NOT invoked.
3. Subtask statuses match expectations.
4. A non-empty final answer was produced.

Results are persisted to `eval_runs` so Task 4C can chart pass-rate over time.

The doubles in [eval/doubles.py](eval/doubles.py) are wired through the registry — never via `unittest.mock.patch`. That choice is deliberate: a real production agent is replaced by a test variant in one line of DI, mirroring how a real ops team would replace a broken specialist with a fallback.

## What the eval caught (real incident, kept in the log)

On the first real-LLM run, **scenario s2 failed**. Objective: *"What's blocking production today?"*. The planner produced subtasks for `inventory` + `production` and set `needs_report=true` — so the report agent was also dispatched. Result: 4/5 passing, total runtime 61.5s (the wasted calls added latency).

**Diagnosis:** the planner prompt only had a negative example in one direction ("if only about stock, do not call production"). It had no inverse rule and nothing forbidding `needs_report` for single-topic questions. Llama 3.3 70B defaulted to "be helpful, gather more context" — over-routing.

**Fix:** rewrote `PLANNER_SYSTEM` in [agents/supervisor.py](agents/supervisor.py) with:
- Bidirectional negative examples ("only production -> only production; do NOT call inventory; do NOT include a report").
- An explicit rule that `needs_report` MUST be false unless the user used the word briefing/report/summary.
- A counter-rule that "complex" or "important" is NOT a reason to add agents.

**Result after fix:** 5/5 passing in 12.7s — a 5x speedup as a side benefit of dropping the spurious calls.

**Tradeoff this exposed (also added to ARCHITECTURE.md):** prompt-engineered routing is fragile to model upgrades. The upgrade path is to add a routing-classifier eval suite (3-5 cases per agent) that runs nightly and alerts on regressions when the underlying model rolls forward.
